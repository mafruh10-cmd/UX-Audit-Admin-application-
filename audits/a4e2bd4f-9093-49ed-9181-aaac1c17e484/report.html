Accessibility
WCAG 2.2 AA
The Glassdoor job search screen demonstrates a functional split-panel layout with reasonable information hierarchy, but suffers from several accessibility violations including icon-only interactive buttons, low-contrast placeholder text, and ambiguous qualification toggle controls. The embedded 'Enhance your jobs' salary widget introduces significant cognitive interruption mid-scroll within the job list, and the dual checkmark/X qualification UI lacks sufficient labeling for assistive technology users.

Detailed Findings
Issue 01WCAG 4.1.2 · A
Qualification toggle controls are icon-only with no accessible text label
📍 Qualifications panel — checkmark and X icon pairs beside each skill (e.g. 'Presentation creation', 'Team development')
High
The Problem
Each qualification row shows a checkmark icon and an X icon with no visible text label or discernible button name, making it impossible for screen reader users to understand what each control does or which skill it applies to. The purpose of toggling between check and X is also visually ambiguous — it is unclear whether these are current matches, user-confirmable skills, or filters. This directly violates WCAG 2.2 SC 1.1.1 (non-text content must have a text alternative) and SC 4.1.2 (name, role, value must be programmatically determinable).
Why critical: Icon-only controls with no accessible name violate WCAG 2.2 SC 4.1.2, breaking assistive technology compatibility entirely for this feature. Per The Design of Everyday Things (Norman), the Gulf of Execution is wide when users cannot determine what an action will do before committing to it.
Recommendation
Add aria-label attributes to each button that include both the action and the skill name (e.g. 'Confirm Presentation creation skill', 'Remove Presentation creation skill'). Supplement icons with visible micro-labels or a legend explaining the toggle meaning. Consider replacing the dual-icon toggle with a single checkbox per skill using a clear 'I have this skill' label.
Reference: WCAG 2.2 — 4.1.2 Name, Role, Value (A): all UI components must expose name, role, and value to assistive technologies
Issue 02WCAG 1.3.1 · A
Salary input field relies solely on placeholder text with no persistent visible label
📍 'Desired minimum salary' text input — dollar sign placeholder inside the field
High
The Problem
The salary input field uses only a '$' symbol as a placeholder with no persistent visible label above or beside it, meaning the label disappears the moment a user begins typing. This fails WCAG 2.2 SC 1.3.5 (Identify Input Purpose) and SC 2.4.6 (Headings and Labels), and forces users to rely on working memory to recall what they were entering. Users with cognitive disabilities or those using autofill will have no programmatic label to reference.
Why critical: Placeholder-only inputs violate WCAG 2.2 SC 1.3.1 (Info and Relationships) because the visible label is not persistently associated with the field in the DOM. This also creates high cognitive load per Psych 101 (Kleinman) — working memory limitations mean disappearing labels force unnecessary recall effort.
Recommendation
Add a persistent visible text label ('Minimum annual salary') positioned above the input field that remains visible during and after input. Ensure the label is programmatically associated via a 'for'/'id' pair or aria-labelledby. The '$' symbol should be retained as a prefix adornment, not as the sole label.
Reference: WCAG 2.2 — 1.3.1 Info and Relationships (A): information conveyed through visual formatting must be programmatically determinable
Issue 03WCAG 4.1.2 · A
Bookmark icon button on job detail panel has no accessible label
📍 Job detail panel — bookmark/save icon button adjacent to 'Apply on employer site' CTA
High
The Problem
The bookmark icon button in the job detail header area provides no visible text label and almost certainly lacks an aria-label, making its purpose opaque to screen reader users and visually ambiguous to sighted users who may not recognise the save/bookmark metaphor. This violates WCAG 2.2 SC 1.1.1 and SC 4.1.2. Users cannot confidently distinguish it from a decorative element.
Why critical: WCAG 2.2 SC 4.1.2 requires that all interactive UI components have a programmatically determinable name. Per The Design of Everyday Things (Norman) — Affordance principle, an icon button must communicate its function through visual and programmatic cues, not icon recognition alone.
Recommendation
Add aria-label='Save job' to the bookmark button and consider adding a visible tooltip on hover/focus. On mobile breakpoints, add a visible 'Save' text label alongside the icon. Ensure the button meets the 24×24 px minimum touch target size per WCAG 2.2 SC 2.5.8.
Reference: WCAG 2.2 — 4.1.2 Name, Role, Value (A): icon-only buttons must have a programmatic accessible name
Issue 04
Inline salary widget interrupts job list scan flow causing severe cognitive disruption
📍 'Enhance your jobs — Desired minimum salary' card injected between job listings in the left panel
High
The Problem
A salary preference input card is embedded between the second and third job listing, breaking the visual rhythm users expect when scanning a list of results. This forces a context switch mid-scan, disrupting the F-pattern reading flow and creating a false impression of fewer job results. Users may mistake it for a job listing or become confused about how many real results exist.
Why critical: Per Don't Make Me Think (Krug) — Billboard Design 101, users scan lists in predictable patterns and unexpected interruptions force re-orientation, dramatically increasing cognitive load. Hick's Law (Psychology of Design) also applies — injecting an unrelated decision task mid-scan adds unnecessary choice complexity at the wrong moment.
Recommendation
Move the salary preference widget to a dedicated onboarding flow, a profile settings page, or a collapsible sidebar panel — never inject it inline within search results. If inline placement is retained, visually differentiate it with a full-width banner treatment clearly labelled 'Personalize your search' that is obviously non-job content. Add a prominent persistent dismiss control.
Reference: Don't Make Me Think (Krug) — Billboard Design 101: users scan, not read; interruptions in predictable patterns force costly re-orientation
Issue 05WCAG 1.4.3 · AA
Low contrast on secondary metadata text in job listing cards
📍 Job listing cards — location and salary text (e.g. 'Berkeley, CA', '$120K – $140K [Employer provided]')
Medium
The Problem
The location and salary metadata text in the job listing cards appears in a light grey tone against the white card background. For normal-weight text at approximately 12–13px, this combination is likely to fall below the WCAG 4.5:1 contrast ratio requirement, making these critical decision-making data points difficult to read for users with low vision or in bright ambient lighting. Salary range is arguably the most important scan-point for job seekers.
Why critical: WCAG 2.2 SC 1.4.3 requires a minimum 4.5:1 contrast ratio for normal text. Per Designing User Interfaces (Malewicz), contrast is the primary tool for establishing hierarchy — reducing contrast on high-value content like salary inverts the intended information priority.
Recommendation
Increase the grey value of metadata text to achieve at minimum a 4.5:1 contrast ratio against white (approximately #767676 or darker). Verify all secondary text across card variants using a contrast checker tool such as the WebAIM Contrast Checker. Consider bolding or slightly enlarging salary figures to reinforce their decision-making importance.
Reference: WCAG 2.2 — 1.4.3 Contrast Minimum (AA): normal text requires 4.5:1 contrast ratio against its background
Issue 06
'Get insights' CTA competes with 'Apply on employer site' primary action creating hierarchy confusion
📍 Job detail panel — 'Get insights' green button in the resume match section vs. 'Apply on employer site' primary CTA
Medium
The Problem
The 'Get insights' button uses the same dark green filled style as the primary 'Apply on employer site' button, creating two visually equivalent primary calls-to-action on the same screen. This violates the principle of a single dominant action per view and creates decision paralysis for users ready to apply. The eye does not naturally know which action to take first.
Why critical: Per Refactoring UI — use color to support hierarchy, not communicate it alone. Having two identically styled primary buttons undermines visual hierarchy and violates Hick's Law (Psychology of Design) — presenting equally weighted competing actions slows decision-making and increases abandonment risk.
Recommendation
Demote 'Get insights' to a secondary button style (outlined or ghost variant) to establish clear visual hierarchy with 'Apply on employer site' as the unambiguous primary action. Consider repositioning the resume match section below the qualifications panel so the apply CTA is encountered first without distraction.
Reference: Refactoring UI — Use color to support hierarchy, not communicate it alone; one primary action per view prevents decision paralysis
Issue 07
Dual search bar layout creates unclear entry-point hierarchy for job search
📍 Top search area — two side-by-side input fields ('Find your perfect job' and 'City, state, zipcode, or remote')
Medium
The Problem
The two search fields sit at the top of the page without a visible submit button, relying on users knowing to press Enter or click elsewhere to initiate a search. New users may be uncertain whether to fill both fields or just one, and whether a search has already been executed given results are already showing below. The absence of an explicit Search/Find Jobs button reduces perceived affordance.
Why critical: Per The Design of Everyday Things (Norman) — Signifier principle, interactive elements must visibly communicate how to use them. A search form without a visible submit button creates a Gulf of Execution, particularly for less tech-savvy users or those using keyboard-only navigation who may not know to press Enter.
Recommendation
Add a visible 'Find Jobs' submit button to the right of the location input field. Include a brief label or active-state indicator showing when a search is live (e.g. 'Showing results for Designer jobs near Berkeley, CA'). Ensure the search form is wrapped in a semantic form element with an accessible submit action.
Reference: The Design of Everyday Things (Norman) — Signifier: visible cues must indicate where and how to act; absent submit button creates Gulf of Execution
Issue 08WCAG 2.5.8 · AA
'Show more' qualifications link has insufficient touch target size
📍 Qualifications panel — 'Show more ˅' text link at the bottom of the qualifications list
Low
The Problem
The 'Show more' control appears as a small inline text link with a chevron icon, likely below the 24×24 CSS pixel minimum touch target required by WCAG 2.2 SC 2.5.8. On touch devices this makes the control difficult to activate without accidental misses, particularly for users with motor impairments or larger fingers. The chevron-only variant at small sizes compounds the problem.
Why critical: WCAG 2.2 SC 2.5.8 (Minimum Target Size, AA) requires interactive targets to be at least 24×24 CSS pixels. Per Universal Principles of Design — Fitts' Law, small distant targets take disproportionately longer to acquire, increasing error rates and frustration.
Recommendation
Increase the clickable area of the 'Show more' control to a minimum of 44×44 px using padding rather than increasing the visual text size. Ensure the control is implemented as a button element (not a div or span) so it is keyboard-focusable by default. Add a visible focus ring meeting WCAG 2.2 SC 2.4.11 requirements.
Reference: WCAG 2.2 — 2.5.8 Minimum Target Size (AA): interactive targets must meet a 24×24 CSS pixel minimum
Issue 09
Success notification banner lacks sufficient urgency differentiation and dismiss affordance clarity
📍 Top of left panel — green success banner 'Great! You're on your way'
Low
The Problem
The success banner uses a light green background that blends closely with the overall green brand palette of the page, reducing its ability to stand out as a distinct system status message. The dismiss X icon is small and may not meet minimum touch target sizing. Additionally, the banner persists visually without indicating whether it will auto-dismiss, leaving users uncertain if action is required.
Why critical: Per The Design of Everyday Things (Norman) — Feedback principle, every system status message must produce a clear, immediate, and unambiguous response that users can distinguish from static content. When status messages visually merge with the surrounding UI, the Gulf of Evaluation widens.
Recommendation
Add a visible border or left-side accent stripe to the success banner to differentiate it from content cards despite shared green coloring. Increase the dismiss button's touch target to at least 44×44 px and add an aria-label='Dismiss notification'. If the banner auto-dismisses, add a visible countdown or progress indicator so users understand its transience.
Reference: The Design of Everyday Things (Norman) — Feedback: every action must produce an immediate, clear, and distinguishable response
