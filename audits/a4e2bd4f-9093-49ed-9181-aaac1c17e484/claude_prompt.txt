AUDIT REPORT  —  Job Search Results & Job Detail Panel
====================================================================
Overall Score:        6.4/10  —  Good
Accessibility Score:  5.5/10  (WCAG 2.2 AA)
Issues Found:         9  (4 High · 3 Medium · 2 Low)

EXECUTIVE SUMMARY
The Glassdoor job search screen demonstrates a functional split-panel layout with reasonable information hierarchy, but suffers from several accessibility violations including icon-only interactive buttons, low-contrast placeholder text, and ambiguous qualification toggle controls. The embedded 'Enhance your jobs' salary widget introduces significant cognitive interruption mid-scroll within the job list, and the dual checkmark/X qualification UI lacks sufficient labeling for assistive technology users.

====================================================================
ISSUES TO FIX  (all issues must be addressed — ordered by severity)
====================================================================

[01] HIGH  —  Qualification toggle controls are icon-only with no accessible text label
     Location:       Qualifications panel — checkmark and X icon pairs beside each skill (e.g. 'Presentation creation', 'Team development')
     Problem:        Each qualification row shows a checkmark icon and an X icon with no visible text label or discernible button name, making it impossible for screen reader users to understand what each control does or which skill it applies to. The purpose of toggling between check and X is also visually ambiguous — it is unclear whether these are current matches, user-confirmable skills, or filters. This directly violates WCAG 2.2 SC 1.1.1 (non-text content must have a text alternative) and SC 4.1.2 (name, role, value must be programmatically determinable).
     Fix:            Add aria-label attributes to each button that include both the action and the skill name (e.g. 'Confirm Presentation creation skill', 'Remove Presentation creation skill'). Supplement icons with visible micro-labels or a legend explaining the toggle meaning. Consider replacing the dual-icon toggle with a single checkbox per skill using a clear 'I have this skill' label.
     Reference:      WCAG 2.2 — 4.1.2 Name, Role, Value (A): all UI components must expose name, role, and value to assistive technologies
     WCAG:           4.1.2  —  Level A

[02] HIGH  —  Salary input field relies solely on placeholder text with no persistent visible label
     Location:       'Desired minimum salary' text input — dollar sign placeholder inside the field
     Problem:        The salary input field uses only a '$' symbol as a placeholder with no persistent visible label above or beside it, meaning the label disappears the moment a user begins typing. This fails WCAG 2.2 SC 1.3.5 (Identify Input Purpose) and SC 2.4.6 (Headings and Labels), and forces users to rely on working memory to recall what they were entering. Users with cognitive disabilities or those using autofill will have no programmatic label to reference.
     Fix:            Add a persistent visible text label ('Minimum annual salary') positioned above the input field that remains visible during and after input. Ensure the label is programmatically associated via a 'for'/'id' pair or aria-labelledby. The '$' symbol should be retained as a prefix adornment, not as the sole label.
     Reference:      WCAG 2.2 — 1.3.1 Info and Relationships (A): information conveyed through visual formatting must be programmatically determinable
     WCAG:           1.3.1  —  Level A

[03] HIGH  —  Bookmark icon button on job detail panel has no accessible label
     Location:       Job detail panel — bookmark/save icon button adjacent to 'Apply on employer site' CTA
     Problem:        The bookmark icon button in the job detail header area provides no visible text label and almost certainly lacks an aria-label, making its purpose opaque to screen reader users and visually ambiguous to sighted users who may not recognise the save/bookmark metaphor. This violates WCAG 2.2 SC 1.1.1 and SC 4.1.2. Users cannot confidently distinguish it from a decorative element.
     Fix:            Add aria-label='Save job' to the bookmark button and consider adding a visible tooltip on hover/focus. On mobile breakpoints, add a visible 'Save' text label alongside the icon. Ensure the button meets the 24×24 px minimum touch target size per WCAG 2.2 SC 2.5.8.
     Reference:      WCAG 2.2 — 4.1.2 Name, Role, Value (A): icon-only buttons must have a programmatic accessible name
     WCAG:           4.1.2  —  Level A

[04] HIGH  —  Inline salary widget interrupts job list scan flow causing severe cognitive disruption
     Location:       'Enhance your jobs — Desired minimum salary' card injected between job listings in the left panel
     Problem:        A salary preference input card is embedded between the second and third job listing, breaking the visual rhythm users expect when scanning a list of results. This forces a context switch mid-scan, disrupting the F-pattern reading flow and creating a false impression of fewer job results. Users may mistake it for a job listing or become confused about how many real results exist.
     Fix:            Move the salary preference widget to a dedicated onboarding flow, a profile settings page, or a collapsible sidebar panel — never inject it inline within search results. If inline placement is retained, visually differentiate it with a full-width banner treatment clearly labelled 'Personalize your search' that is obviously non-job content. Add a prominent persistent dismiss control.
     Reference:      Don't Make Me Think (Krug) — Billboard Design 101: users scan, not read; interruptions in predictable patterns force costly re-orientation

[05] MEDIUM  —  Low contrast on secondary metadata text in job listing cards
     Location:       Job listing cards — location and salary text (e.g. 'Berkeley, CA', '$120K – $140K [Employer provided]')
     Problem:        The location and salary metadata text in the job listing cards appears in a light grey tone against the white card background. For normal-weight text at approximately 12–13px, this combination is likely to fall below the WCAG 4.5:1 contrast ratio requirement, making these critical decision-making data points difficult to read for users with low vision or in bright ambient lighting. Salary range is arguably the most important scan-point for job seekers.
     Fix:            Increase the grey value of metadata text to achieve at minimum a 4.5:1 contrast ratio against white (approximately #767676 or darker). Verify all secondary text across card variants using a contrast checker tool such as the WebAIM Contrast Checker. Consider bolding or slightly enlarging salary figures to reinforce their decision-making importance.
     Reference:      WCAG 2.2 — 1.4.3 Contrast Minimum (AA): normal text requires 4.5:1 contrast ratio against its background
     WCAG:           1.4.3  —  Level AA

[06] MEDIUM  —  'Get insights' CTA competes with 'Apply on employer site' primary action creating hierarchy confusion
     Location:       Job detail panel — 'Get insights' green button in the resume match section vs. 'Apply on employer site' primary CTA
     Problem:        The 'Get insights' button uses the same dark green filled style as the primary 'Apply on employer site' button, creating two visually equivalent primary calls-to-action on the same screen. This violates the principle of a single dominant action per view and creates decision paralysis for users ready to apply. The eye does not naturally know which action to take first.
     Fix:            Demote 'Get insights' to a secondary button style (outlined or ghost variant) to establish clear visual hierarchy with 'Apply on employer site' as the unambiguous primary action. Consider repositioning the resume match section below the qualifications panel so the apply CTA is encountered first without distraction.
     Reference:      Refactoring UI — Use color to support hierarchy, not communicate it alone; one primary action per view prevents decision paralysis

[07] MEDIUM  —  Dual search bar layout creates unclear entry-point hierarchy for job search
     Location:       Top search area — two side-by-side input fields ('Find your perfect job' and 'City, state, zipcode, or remote')
     Problem:        The two search fields sit at the top of the page without a visible submit button, relying on users knowing to press Enter or click elsewhere to initiate a search. New users may be uncertain whether to fill both fields or just one, and whether a search has already been executed given results are already showing below. The absence of an explicit Search/Find Jobs button reduces perceived affordance.
     Fix:            Add a visible 'Find Jobs' submit button to the right of the location input field. Include a brief label or active-state indicator showing when a search is live (e.g. 'Showing results for Designer jobs near Berkeley, CA'). Ensure the search form is wrapped in a semantic form element with an accessible submit action.
     Reference:      The Design of Everyday Things (Norman) — Signifier: visible cues must indicate where and how to act; absent submit button creates Gulf of Execution

[08] LOW  —  'Show more' qualifications link has insufficient touch target size
     Location:       Qualifications panel — 'Show more ˅' text link at the bottom of the qualifications list
     Problem:        The 'Show more' control appears as a small inline text link with a chevron icon, likely below the 24×24 CSS pixel minimum touch target required by WCAG 2.2 SC 2.5.8. On touch devices this makes the control difficult to activate without accidental misses, particularly for users with motor impairments or larger fingers. The chevron-only variant at small sizes compounds the problem.
     Fix:            Increase the clickable area of the 'Show more' control to a minimum of 44×44 px using padding rather than increasing the visual text size. Ensure the control is implemented as a button element (not a div or span) so it is keyboard-focusable by default. Add a visible focus ring meeting WCAG 2.2 SC 2.4.11 requirements.
     Reference:      WCAG 2.2 — 2.5.8 Minimum Target Size (AA): interactive targets must meet a 24×24 CSS pixel minimum
     WCAG:           2.5.8  —  Level AA

[09] LOW  —  Success notification banner lacks sufficient urgency differentiation and dismiss affordance clarity
     Location:       Top of left panel — green success banner 'Great! You're on your way'
     Problem:        The success banner uses a light green background that blends closely with the overall green brand palette of the page, reducing its ability to stand out as a distinct system status message. The dismiss X icon is small and may not meet minimum touch target sizing. Additionally, the banner persists visually without indicating whether it will auto-dismiss, leaving users uncertain if action is required.
     Fix:            Add a visible border or left-side accent stripe to the success banner to differentiate it from content cards despite shared green coloring. Increase the dismiss button's touch target to at least 44×44 px and add an aria-label='Dismiss notification'. If the banner auto-dismisses, add a visible countdown or progress indicator so users understand its transience.
     Reference:      The Design of Everyday Things (Norman) — Feedback: every action must produce an immediate, clear, and distinguishable response
