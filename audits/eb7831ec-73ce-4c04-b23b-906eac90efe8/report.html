Detailed Findings
Issue 01WCAG 1.4.5 · AA
Content calendar is rendered as an image of text
📍 Main content area — Content Calendar table for March 2026
High
The Problem
The entire content calendar table appears to be rendered as an image rather than semantic HTML. Screen readers cannot parse the table structure, rows, columns, or cell content. Users cannot select, copy, or interact with individual cell text.
Why critical: Images of text violate WCAG 1.4.5 because users cannot resize, reflow, or customize the text presentation, and assistive technologies cannot access the content programmatically.
Recommendation
Render the calendar as a semantic HTML with proper rows, and
headers,
cells. Include a
element describing the table. This also enables responsive behavior on smaller screens.
Reference: WCAG 2.2 — 1.4.5 Images of Text (AA): text should be used to convey information rather than images of text
Issue 02
Multiple spelling errors in AI-generated calendar content
📍 Content Calendar table cells — 'Photd', 'Ive scener', 'Tentmonial', 'Tucks', 'Triday'
High
The Problem
The calendar contains at least five typographical errors: 'Photd' (Photo), 'Behind Ive scener' (Behind the scenes), 'Tentmonial' (Testimonial), 'Tips and Tucks' (Tips and Tricks), and 'Triday' (Friday). These errors severely undermine trust in the AI tool and the product's professionalism.
Why critical: Per the aesthetic-usability effect, users perceive sloppy outputs as less trustworthy and functional. Errors in an AI content tool directly contradict the product's core value proposition, causing users to question its reliability.
Recommendation
Implement spell-checking and content validation on AI-generated outputs before rendering. Display a disclaimer or allow inline editing of generated content. Consider a review step before finalizing the calendar.
Reference: Universal Principles of Design — Aesthetic-Usability Effect: beautiful things are perceived as easier to use
Issue 03WCAG 1.1.1 · A
Download and thumbs-up icon buttons lack accessible labels
📍 Below the calendar table — download icon and thumbs-up icon
High
The Problem
The two icon-only buttons beneath the calendar (download and like/feedback) have no visible text labels or apparent aria-labels. Screen reader users cannot determine the purpose of these interactive controls.
Why critical: WCAG 1.1.1 requires all non-text content that is functional to have a text alternative. Icon-only buttons without labels force users to guess functionality, increasing the gulf of execution.
Recommendation
Add aria-label attributes such as 'Download calendar' and 'Rate this response' to each button. Consider adding visible tooltip text on hover/focus to aid sighted users as well.
Reference: WCAG 2.2 — 1.1.1 Non-text Content (A): all non-text content must have a text alternative
Issue 04
Tag 'salmobbin' appears to be a misspelling or unclear label
📍 Chat input area — 'salmobbin' tag chip
Medium
The Problem
The tag 'salmobbin' in the input field appears to be a misspelled or unclear label that provides no recognizable meaning to the user. Users cannot understand what this tag represents or why it was applied, creating confusion about the input context.
Why critical: Per Norman's signifier principle, visible cues must clearly indicate their meaning and purpose. An unrecognizable label increases cognitive load and forces users to investigate or guess its function.
Recommendation
Ensure tag labels are human-readable and meaningful (e.g., a brand name or content category). If auto-generated, validate against known terms. Provide a tooltip explaining the tag's purpose on hover.
Reference: The Design of Everyday Things (Norman) — Signifier: visible cues must indicate where to act
Issue 05WCAG 1.4.3 · AA
Insufficient contrast on table header text against white background
📍 Content Calendar table header row — 'Monday', 'Tuesday', etc.
Medium
The Problem
The bold header text in the table uses a dark color on a white/light gray background, but some cell text within the table body may not meet the 4.5:1 minimum contrast ratio, especially the lighter-weight body text. The overall table rendering as an image also prevents zoom-based contrast improvements.
Why critical: WCAG 1.4.3 requires a minimum contrast ratio of 4.5:1 for normal text. Insufficient contrast impacts users with low vision and those viewing screens in bright environments.
Recommendation
Ensure all table text meets a minimum 4.5:1 contrast ratio against its background. Convert the image-based table to HTML text so users can apply custom styles or zoom without degradation.
Reference: WCAG 2.2 — 1.4.3 Contrast (Minimum) (AA): normal text requires 4.5:1 contrast ratio
Issue 06WCAG 2.4.7 · AA
No visible focus indicators on interactive elements
📍 Sidebar navigation items, input field, icon buttons
Medium
The Problem
There are no visible focus indicators on any interactive elements in the screenshot. Keyboard users navigating through the sidebar, input field, or action buttons would have no visual indication of which element is currently focused.
Why critical: WCAG 2.4.7 requires that keyboard focus indicators are visible. Without them, keyboard-only users cannot orient themselves on the page, making the interface effectively unusable without a mouse.
Recommendation
Add a visible focus ring (e.g., 2px solid blue outline with offset) to all focusable elements. Ensure the focus indicator has at least a 3:1 contrast ratio against adjacent colors per WCAG 2.4.13.
Reference: WCAG 2.2 — 2.4.7 Focus Visible (AA): keyboard focus indicator must be visible
Issue 07
Calendar lacks any editing or interaction affordances
📍 Content Calendar table area
Medium
The Problem
The generated calendar appears to be a static image with no way for users to edit individual cells, rearrange content types, or customize the output. Users must regenerate the entire calendar to make any changes, creating an inefficient workflow.
Why critical: Per Norman's gulf of execution, users who want to tweak one cell cannot figure out how to do so. This forces a costly regeneration cycle and violates the principle of user control and freedom (Nielsen's Heuristic #3).
Recommendation
Make the calendar an editable table where users can click cells to modify content types. Add drag-and-drop functionality for rearranging items. Provide an 'Edit' button that converts the static view into an editable mode.
Reference: The Design of Everyday Things (Norman) — Gulf of Execution: user cannot figure out how to do what they want
Issue 08WCAG 1.3.1 · A
Input placeholder text disappears without persistent label
📍 Chat input field — 'Describe the image you want to create'
Medium
The Problem
The placeholder text 'Describe the image you want to create' serves as the only label for the input field. Once a user begins typing, this instructional text disappears, leaving no persistent label to remind users of the field's purpose. Additionally, the placeholder says 'image' but the feature generates content calendars, creating a mismatch.
Why critical: Placeholder-only labels violate WCAG 1.3.1 because the programmatic relationship between label and input is lost. The label/purpose mismatch also increases cognitive dissonance and user confusion.
Recommendation
Add a persistent visible label above the input field (e.g., 'Ask OwlyGPT'). Update the placeholder to accurately reflect the tool's capabilities: 'Describe what you want to create — calendars, posts, images, and more.'
Reference: WCAG 2.2 — 1.3.1 Info and Relationships (A): information and relationships conveyed visually must be programmatically determinable
Issue 09WCAG 1.4.1 · A
Sidebar active state relies primarily on color/fill change
📍 Left sidebar — OwlyGPT nav item (highlighted with dark background)
Low
The Problem
The active navigation item (OwlyGPT) is indicated primarily through a filled dark background circle, while inactive items use outline-style icons. The differentiation is subtle and may not be perceivable by users with color vision deficiencies or low vision.
Why critical: Per WCAG 1.4.1, color should not be the sole means of conveying information. A subtle fill change may be missed, leaving users uncertain about their current location in the app.
Recommendation
Add a secondary indicator such as a left border accent bar, bold text label, or distinct shape change to reinforce the active state. Ensure at least 3:1 contrast between active and inactive states.
Reference: WCAG 2.2 — 1.4.1 Use of Color (A): color is not the only visual means of conveying information
Issue 10
No confirmation or success feedback after download action
📍 Download icon button below the calendar
Low
The Problem
The download button provides no visible indication of what will happen when clicked (file format, destination) and likely provides no confirmation feedback after the action is completed. Users cannot tell what they're downloading or if it succeeded.
Why critical: Norman's feedback principle states every action must produce an immediate, clear response. Without feedback, users may click repeatedly or assume the action failed, eroding confidence in the tool.
Recommendation
Add a tooltip showing 'Download as PNG' (or relevant format) on hover. After clicking, display a brief toast notification confirming 'Calendar downloaded successfully.' Consider offering format options (PNG, CSV, PDF).
Reference: The Design of Everyday Things (Norman) — Feedback: every action must produce an immediate, clear response
