However, several accessibility issues around color contrast, color-only status indicators, and truncated content hurt usability. The information density is high but could benefit from better visual hierarchy and clearer interactive affordances.

Detailed Findings
Issue 01WCAG 1.4.1 · A
Status badges rely on color alone to convey meaning
📍 Status: Active badges on the project cards (top row)
High
The Problem
The 'Active' status is conveyed solely through a green color badge with no supporting icon or pattern. Users with color vision deficiencies cannot distinguish active from other statuses. This affects all four project cards in the top row.
Why critical: WCAG 1.4.1 requires that color is not the sole means of conveying information. Approximately 8% of males have color vision deficiency, making this a significant accessibility barrier.
Recommendation
Add a secondary indicator such as an icon (e.g., a checkmark or dot with a border pattern) alongside the color. Consider adding a distinct shape or text styling treatment so status is perceivable without color.
Reference: WCAG 2.2 — 1.4.1 Use of Color (A): information must not be conveyed by color alone
Issue 02WCAG 1.4.3 · AA
Low contrast on secondary text throughout dashboard
📍 Gray metadata text such as 'Created by Alex Smith', 'Collections added', 'Not shared • 6 minutes ago • 1 attribute'
High
The Problem
Multiple secondary text elements appear in a light gray that likely fails the 4.5:1 contrast ratio against the white background. This includes creator attribution, descriptions on cards, and metric subtitles. This makes the text difficult to read for users with low vision or in bright environments.
Why critical: WCAG 1.4.3 mandates a minimum 4.5:1 contrast ratio for normal-sized text. The light gray (~#999 or lighter) on white falls below this threshold, causing readability issues.
Recommendation
Increase the text color to at least #767676 or darker to achieve the 4.5:1 minimum contrast ratio. Verify all secondary text elements with a contrast checker tool and establish a design token for secondary text that meets AA requirements.
Reference: WCAG 2.2 — 1.4.3 Contrast Minimum (AA): normal text requires 4.5:1 contrast ratio
Issue 03
Truncated card descriptions hide critical information
📍 Project cards in the top row — 'Looking for a Senior Software Engineer with 5...' and 'Product Designer with 3+ years of experience i...'
High
The Problem
Card descriptions are truncated mid-word with no tooltip or expand mechanism visible. Users cannot access the full search criteria without clicking through, which forces unnecessary navigation and increases cognitive load when comparing projects.
Why critical: Per the information scent principle, users need enough visible detail to decide whether to pursue an item. Truncation without progressive disclosure creates a Gulf of Evaluation where users cannot assess relevance from the overview.
Recommendation
Add a tooltip on hover showing the full description text, or increase the card height to accommodate 2-3 lines. Alternatively, provide an expand/collapse mechanism inline to allow quick scanning without navigation.
Reference: The Design of Everyday Things (Norman) — Gulf of Evaluation: user cannot tell if their action worked
Issue 04WCAG 2.4.7 · AA
No visible focus indicators for interactive elements
📍 All interactive elements — navigation links, project cards, candidate rows, buttons
High
The Problem
There are no visible focus indicators shown on any interactive elements in the interface. Keyboard-only users cannot determine which element currently has focus, making the dashboard essentially unusable without a mouse.
Why critical: WCAG 2.4.7 requires a visible focus indicator on all interactive elements. Without this, keyboard users and assistive technology users cannot navigate the interface, creating a complete barrier to access.
Recommendation
Implement a visible focus ring (at least 2px solid outline in a contrasting color) on all focusable elements including cards, links, buttons, and list items. Use CSS :focus-visible to avoid showing focus rings on mouse clicks while maintaining them for keyboard navigation.
Reference: WCAG 2.2 — 2.4.7 Focus Visible (AA): keyboard focus indicator must be visible
Issue 05WCAG 1.1.1 · A
Bar chart lacks text alternatives and axis labels
📍 Profiles added today bar chart (right side of dashboard)
Medium
The Problem
The bar chart showing '106 Profiles added today' has minimal axis labeling — only dates on the x-axis with no y-axis values. Screen reader users would have no way to interpret the data, and sighted users cannot determine exact values for each bar.
Why critical: WCAG 1.1.1 requires non-text content to have text alternatives. Charts without alt text or data tables are inaccessible to screen reader users and violate the principle of equivalent access to information.
Recommendation
Provide an accessible data table as an alternative or use ARIA labels to describe the chart data. Add y-axis labels with values, and implement tooltips on hover showing exact daily counts for each bar.
Reference: WCAG 2.2 — 1.1.1 Non-text Content (A): all non-text content must have a text alternative
Issue 06
Candidate list items lack clear clickable affordance
📍 Candidate list on the left side (Claire G., Robert Barker, etc.)
Medium
The Problem
The candidate rows show a subtle chevron (>) on the right but lack hover states, underlines, or other conventional affordances to indicate interactivity. Users may not immediately recognize these as clickable elements, reducing discoverability.
Why critical: Per Norman's signifier principle, interactive elements must clearly communicate that they are actionable. Subtle chevrons alone may not provide sufficient visual cues, especially for less tech-savvy users, increasing the Gulf of Execution.
Recommendation
Add a visible hover state with a background color change or subtle elevation. Consider adding a text link style (e.g., 'View profile') or make the entire row more obviously interactive with cursor changes and visual feedback on hover.
Reference: The Design of Everyday Things (Norman) — Signifier: visible cues must indicate where to act
Issue 07
Metrics section lacks contextual meaning and actionability
📍 Summary stats row — Total Lists (3), Candidates Saved (203), Total Searches (9), Candidates Outreached (0)
Medium
The Problem
The four metric cards present raw numbers without trends, goals, or comparisons. The '0 No emails sent yet' metric provides no guidance on next steps. Users cannot determine if these numbers represent good or poor performance without context.
Why critical: According to the signal-to-noise ratio principle, data without context is noise. Users need comparative benchmarks or trend indicators to extract actionable insights, otherwise the metrics occupy prime visual real estate without delivering value.
Recommendation
Add trend indicators (up/down arrows with percentages), comparison to previous period, or progress toward a goal. For the '0 emails' metric, add a CTA like 'Start your first outreach campaign' to guide the user toward meaningful action.
Reference: Universal Principles of Design — Signal-to-Noise Ratio: maximise relevant information, minimise clutter
Issue 08
Trial notification competes with primary navigation
📍 Bottom-left trial banner — 'Your Wrangle trial is active'
Medium
The Problem
The trial notification is placed directly above important navigation items (Invite team, Knowledge hub, Account settings), creating visual clutter and potentially obscuring navigation on smaller screens. The green progress bar draws attention away from the primary dashboard content.
Why critical: According to Gestalt's Law of Proximity, items placed close together are perceived as related. The trial banner's proximity to navigation items may cause users to conflate promotional messaging with functional navigation, increasing cognitive load.
Recommendation
Move the trial notification to a less intrusive location such as a top banner or integrate it into Account settings. Alternatively, make it dismissible so users can remove it once acknowledged. Reduce visual weight by using a subtler design.
Reference: Psych 101 (Kleinman) — Gestalt: Law of Proximity
Issue 09
Horizontal card scrolling hides content with no visible scroll indicator
📍 Top project cards row — fourth card 'Product Designer' is cut off at the right edge
Medium
The Problem
The fourth project card is truncated at the right edge of the viewport with no visible scroll indicator, pagination dots, or arrow buttons. Users may not realize there are additional cards to view, leading to missed content and incomplete awareness of their projects.
Why critical: The visibility principle states that system status and available actions must be perceivable. Hidden horizontal scrolling violates this by concealing content without visual signifiers, and horizontal scrolling is a non-standard interaction pattern on desktop that many users overlook.
Recommendation
Add left/right navigation arrows at the edges of the card container, or include pagination dots below the cards. Alternatively, show a partial fade effect or 'peek' of the next card to signal scrollability. Consider a 'View all' link if there are many projects.
Reference: Universal Principles of Design — Visibility: system status and available actions must be visible
Issue 10
Inconsistent card metadata labeling across project types
📍 Project cards — 'Sourcing' vs 'Collection' labels and varying metadata formats
Low
The Problem
The four project cards use inconsistent metadata patterns: some show 'Sourcing' labels, one shows 'Collection'; candidates are described as 'shortlisted', 'saved', or with a count; sharing status varies. This inconsistency forces users to parse each card differently.
Why critical: Nielsen's Consistency and Standards heuristic states that similar elements should behave and appear consistently. When the same type of content uses different labeling schemes, users must spend additional cognitive effort interpreting each card individually.
Recommendation
Standardize the card layout so all cards show the same fields in the same order: type badge, description, status, candidate count (with consistent terminology), sharing status, and creator. Use a consistent label like 'X candidates' regardless of their status.
Reference: Designing User Interfaces (Malewicz) — Contrast and visual hierarchy
